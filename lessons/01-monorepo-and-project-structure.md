# Lesson 01 — Monorepo & Project Structure

## What is a Monorepo?

A **monorepo** (short for "mono repository") means keeping all parts of a project — frontend, backend, documentation — inside a single Git repository, rather than splitting them into separate repos.

Our project root looks like this:

```
NextJS-dashboard/           ← single git repo
├── frontend/               ← Next.js app (TypeScript)
├── backend/                ← FastAPI app (Python)
├── lessons/                ← you are here
├── PLAN.md
├── PROJECT.md
├── docker-compose.yml      ← local dev orchestration
└── .gitignore
```

## Why Monorepo?

For a project this size (one frontend, one backend, one developer), a monorepo offers several practical advantages:

| Advantage | What it means in practice |
|---|---|
| **Single source of truth** | One `git clone` gets you everything |
| **Shared history** | A single commit can span both frontend and backend changes |
| **Easier refactoring** | Rename a field in the API schema and update the frontend type in the same commit |
| **Simpler CI/CD** | One pipeline, one set of triggers |
| **No version syncing** | You don't need to coordinate releases between two repos |

The alternative (separate repos) makes sense for large teams where frontend and backend teams are independent. For a personal project, it adds friction with no benefit.

## The `.gitignore` at the Root

We placed `.gitignore` at the **root level**, not inside `/frontend` or `/backend`. This single file covers both:

- **Node artifacts**: `node_modules/`, `.next/`, `out/`, `dist/`
- **Python artifacts**: `__pycache__/`, `.venv/`, `*.egg-info/`
- **Environment files**: `.env`, `.env.local`, `.env.*.local` — **critically important** (see Lesson 08)
- **OS/editor noise**: `.DS_Store`, `.idea/`

Notice that `frontend/` has its own `.gitignore` as well — that was auto-generated by `create-next-app`. The two files don't conflict; Git merges rules from all `.gitignore` files it finds up the directory tree.

## The `docker-compose.yml`

Docker Compose is a tool for running multiple services locally with a single command (`docker compose up`). Our file defines two services:

```yaml
services:
  db:        # local PostgreSQL, mirrors Supabase in prod
  backend:   # FastAPI, hot-reloads from ./backend source files
```

**Why is this useful even if you're using Supabase in production?**

You don't want to run every development test against your live database. The local Postgres container gives you a safe sandbox — blow it away and recreate it anytime with `docker compose down -v && docker compose up`.

The backend service in Compose mounts `./backend:/app` as a volume, which means file edits on your host are instantly reflected inside the container without rebuilding the image.

## Key Takeaway

> The monorepo structure is an **organizational decision**, not a technical one. It shapes your daily workflow: where you `cd`, how you commit, how you deploy. We chose it because simplicity beats flexibility at this project scale.
